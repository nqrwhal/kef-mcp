# Connecting the KEF MCP server to Poke

This folder covers the **Poke-specific** wiring. For building/running the server
itself, see the [root README](../README.md).

```
Poke (cloud) ──Poke tunnel──► kef-mcp (Docker, your always-on host)
                                  ├─ HTTP ─► KEF LSX II (your LAN)
                                  └─ HTTPS ► Spotify Web API
```

Poke is a cloud assistant, so it can't reach a speaker on your LAN directly. The
MCP server runs on an always-on machine at home, and the **official Poke tunnel**
(`poke` CLI) forwards it up to your Poke agent — no port-forwarding, no Cloudflare,
no public hostname.

We run the tunnel in its own container (`poke-tunnel` in `docker-compose.yml`) so it
authenticates headlessly and survives reboots, instead of being a foreground
`poke tunnel` process you have to babysit.

## 1. Get the server config ready

Follow the [root README](../README.md) to fill in `.env` (KEF IP — already
pre-filled to `192.168.4.138`, Spotify app + refresh token). Leave
`MCP_AUTH_TOKEN` **blank** — the Poke tunnel is the trust boundary and doesn't
forward a bearer token to the local server.

## 2. Get your Poke tunnel token

`poke tunnel` authenticates with a **login session token**, not a kitchen API
key. It only reads that token from the credentials file `poke login` writes — so
you log in once on any machine with a browser and copy the token into `.env`.

1. On a machine with a browser (your laptop is fine), run:
   ```bash
   npx poke@latest login
   ```
   This opens a browser device-login flow. Approve it.
2. Copy the `"token"` value out of the credentials file it wrote:
   - macOS/Linux: `~/.config/poke/credentials.json`
   - Windows: `%USERPROFILE%\.config\poke\credentials.json`
3. Put it in `.env` as `POKE_TUNNEL_TOKEN`.
4. Optionally set `POKE_TUNNEL_NAME` (defaults to `KEF + Spotify`) — this is the
   connection name you'll see in Poke.

> Session tokens expire (~1 year). If the tunnel later fails auth, re-run
> `npx poke@latest login` and update `POKE_TUNNEL_TOKEN`.

## 3. Bring it up

```bash
docker compose up -d --build
docker compose logs -f poke-tunnel
```

In the `poke-tunnel` logs you should see it connect and sync tools. The connection
(named `KEF + Spotify`) now shows up in Poke automatically — no manual "add MCP
server" step. Tools re-sync every ~5 minutes while the tunnel runs.

> **Want to test interactively first?** You can run the CLI by hand on any machine:
> ```bash
> npx poke@latest login
> npx poke@latest tunnel http://localhost:8000/mcp --name "KEF + Spotify" --recipe
> ```
> `--recipe` prints a shareable recipe + QR code. But this is a foreground process;
> the `poke-tunnel` container is what you want for an always-on host.

## 4. Add the recipe

Paste the text from [`RECIPE.md`](./RECIPE.md) into Poke as a recipe (or use the one
generated by `--recipe` above). Then just talk to it: *"Play Bonobo on the KEF."*

## Notes

- **Headless auth:** `poke tunnel` reads its account token only from a credentials
  file (`~/.config/poke/credentials.json`) — it does **not** read `POKE_API_KEY`,
  and it wants the session token from `poke login`, not a kitchen API key. The
  container's entrypoint writes that file from `POKE_TUNNEL_TOKEN` at startup, so
  it runs unattended without a browser inside the container.
- The KEF only advertises as a Spotify Connect target once it's awake and has played
  Spotify since boot. `play_on_kef` wakes it first; on a cold boot you may need to
  cast to it from the Spotify app once. The tool returns a clear message if the
  device isn't found.
- If you'd rather expose the server a different way (your own reverse proxy, VPN,
  Cloudflare, etc.), you can — set `MCP_AUTH_TOKEN` and point your client at
  `https://.../mcp` with a Bearer header. The Poke tunnel is just the easiest path.
